## 대표 문제 해결 경험

아래 네 글은 실제로 설계하고 운영한 정산 시스템의 기록입니다. 시스템을 0부터 세운 설계와, 그 시스템이 천만 건 규모로 성장하며 부딪힌 문제를 해결한 트러블슈팅으로 이어집니다. 회사 식별 정보와 사업 지표는 제거하고 기술적 의사결정 과정만 일반화해 정리했습니다.

### 🏗️ 정산 플랫폼 0→1 설계 — 엑셀로 굴러가던 정산을 시스템으로
→ [전체 글 보기](./troubleshooting/settlement-system-zero-to-one.md)

전용 시스템 없이 100% 수동 엑셀로 관리되던 정산을 기획자 1명·백엔드 1명(저) 구성으로 0부터 설계하고 구현했습니다. 출판사(원저작권)와 저자(2차 저작권)가 얽힌 다층 수익 분배가 핵심 난제였습니다. 주문을 정산 기초 데이터로 가공하는 파이프라인을 세우고, 지문 하나에 여러 작품이 매핑되는 1:N 저작권 분배, 이용권의 일할 정산과 이용 점유율 동적 반영, 복합 결제 수단 처리를 모델링했습니다. 정산 오류는 16개 코드로 구조화해 운영자가 개발자 없이 일괄 보정하는 자가복구 구조까지 만들었습니다.

*키워드: 도메인 모델링 · 1:N 정산 알고리즘 · 변하는 것과 변하지 않는 것의 분리 · 운영 자립화*

### 🔍 천만 건 정산 데이터 조회 개선 — 인덱스도 파티셔닝도 안 듣던 이유
→ [전체 글 보기](./troubleshooting/settlement-query-optimization.md)

위에서 만든 정산 시스템이 800만 건을 넘기며 성수기 한 달 조회가 약 19초까지 걸리던 문제입니다. VACUUM·REINDEX·미사용 인덱스 정리, 월별 파티셔닝까지 정공법을 차례로 시도했지만 모두 실패했습니다. EXPLAIN ANALYZE로 확인한 진짜 원인은 프루닝된 단일 파티션의 대량 행과 다중 조인 앞에서 옵티마이저가 인덱스를 버리고 Seq Scan을 택한 것이었습니다. 문제를 쿼리 튜닝에서 데이터 모델로 옮겨, 조인이 0인 조회 전용 Flat Table과 메시지 큐 기반 동기화로 읽기 경로를 분리해 해결했습니다.

*키워드: PostgreSQL 실행 계획 분석 · 파티셔닝의 효과 범위 · 읽기/쓰기 모델 분리 · CQRS·비정규화*

### ⚙️ Bull Queue 워터마크 장애 — 처리 능력은 멀쩡한데 31분을 기다린 이유
→ [전체 글 보기](./troubleshooting/bull-queue-bottleneck.md)

개별 작업은 1~5초면 끝나는데 사용자 대기 시간이 31분까지 치솟은 장애입니다. "작업이 느린 것"과 "큐가 밀리는 것"을 구분하는 데서 출발해 concurrency 미설정이 백로그를 만든 원인임을 짚었습니다. 단순 concurrency 증설로 끝내지 않고 lock/stalled 설정 우선순위 함정, stalled 작업의 멱등성 보장, graceful shutdown, 이식한 오토스케일링 정책의 오작동까지 큐 운영 전반을 재정비한 과정을 담았습니다.

*키워드: Bull Queue 동작 원리 · 멱등성 설계 · Graceful Shutdown · 멀티 인스턴스 분산 락 · 모니터링*

### 🧠 Node.js 워커의 RSS 메모리 누수 — 4일간의 추적기
→ [전체 글 보기](./troubleshooting/redis-job-memory-leak.md)

워커 RSS가 시간당 수백 MB씩 우상향하다 `spawn ENOMEM`으로 죽던 문제입니다. heap·external은 정상인데 RSS만 오르는 상황에서 애플리케이션 코드 → glibc 단편화 → heap 누수 → 잘못된 GC 튜닝 순으로 가설을 기각해 가며 네이티브 라이브러리의 메모리 미반납이라는 진짜 원인에 도달했습니다. 라이브러리 교체 후 메모리 사용률을 약 40%에서 7% 수준으로 낮췄고 처리량이 늘어난 피크 구간에서도 안정적으로 유지됐습니다.

*키워드: Native Memory 추적 · V8 GC와 RSS · glibc malloc arena · 검증 기준 설계*
