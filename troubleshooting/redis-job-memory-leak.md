# Node.js 워커의 RSS 메모리 누수 추적 — heap은 정상인데 ENOMEM이 터진 이유

> 내부 워터마크 처리 작업을 담당하는 Redis 기반 백그라운드 워커에서 RSS 메모리가 시간에 따라 우상향하다 결국 `spawn ENOMEM`으로 죽는 문제를 추적해 해결한 기록입니다. heap이 정상인데 RSS만 오르는 상황에서 가설을 하나씩 데이터로 기각해가며 네이티브 라이브러리가 메모리를 충분히 반환하지 않는다는 진짜 원인에 도달하기까지의 과정을 정리했습니다.

## TL;DR

- **증상**: 워커 컨테이너의 RSS가 시간당 수백 MB씩 누적 → 가상 메모리 확보 실패로 자식 프로세스 fork 시 `ENOMEM` 발생 → 컨테이너 재시작 반복
- **함정**: V8 heap과 external 메모리는 정상 범위였다. 문제는 그 아래 네이티브 레이어에 있었다.
- **결론**: 이미지 처리에 쓰던 `node-canvas`(Cairo 기반 C++ 바인딩)에서 작업이 끝난 뒤에도 네이티브 메모리가 충분히 반환되지 않았다. V8이 관리하는 영역과 분리된 곳에서 RSS가 계속 쌓였고 `@napi-rs/canvas`(Rust/skia)로 교체해 해결했다.
- **결과**: 메모리 사용률 약 40% → 7% 수준으로 하락. 처리량이 기존 대비 55% 늘어난 피크 구간에서도 RSS 수평 유지.

---

## 1. 문제 상황

인프라를 컨테이너 오케스트레이션으로 옮긴 뒤에도 특정 이미지 처리 워커만 RSS 메모리가 꾸준히 우상향하는 문제가 남았습니다. 모니터링상 메모리 사용률이 배포 직후 40% 부근에서 시작해 시간이 갈수록 계속 올랐고 임계에 도달하면 컨테이너가 재시작되는 패턴이 반복됐습니다.

가장 치명적인 형태는 이것이었습니다. RSS가 이미 높은 상태에서 워커가 자식 프로세스를 fork하려 하면 OS가 가상 메모리를 확보하지 못해 `spawn ENOMEM`이 터졌습니다. 메모리 누수가 단순한 리소스 낭비를 넘어 **작업 자체의 실패**로 이어지고 있었습니다.

조건을 정리하면 이렇습니다.

- 8GiB급 인스턴스, 워커 동시성 2, 다중 인스턴스 구성
- 처리량은 시간대별 편차가 크고, 점심시간대 피크가 존재
- heap, external 지표는 겉보기에 정상

마지막 줄이 이 문제를 어렵게 만든 핵심입니다. 보통 Node.js 메모리 누수는 heap에서 잡는데 **heap은 멀쩡한데 RSS만 오르고 있었습니다.**

## 2. 추적: 가설을 세우고 하나씩 기각하기

이 문제 해결의 본질은 정답을 한 번에 맞히는 게 아니라 틀린 가설을 데이터로 빠르게 죽이는 것이었습니다. 4개의 가설을 순서대로 검증했습니다.

### 가설 A — 애플리케이션 코드의 ArrayBuffer 누수

로그를 뜯어보니 특정 임베드 처리 작업에서 `arrayBuffers` 사용량이 비정상적으로 급증하는 패턴이 있었습니다. external 메모리 누수로 판단하고 해당 코드를 수정해 배포했습니다.

**결과: 효과 미미.** arrayBuffers는 정상으로 돌아왔지만 RSS 우상향은 그대로였습니다. 이 단계에서 중요한 사실 하나를 확인했습니다. 애플리케이션 레벨에서 관측되는 heap과 external 지표만으로는 RSS 증가를 설명할 수 없었습니다. 문제는 Node.js가 직접 관리하지 않는 **그 아래 네이티브 레이어**에 있을 가능성이 높았습니다.

### 가설 B — glibc 메모리 단편화

RSS만 오르는 전형적인 원인 중 하나가 glibc의 메모리 단편화(fragmentation)입니다. glibc는 멀티스레드 환경에서 `CPU 수 × 8`개의 arena를 만드는데, arena가 많을수록 각 arena에 반환되지 않은 빈 공간이 흩어져 RSS가 부풀어 오릅니다(메모리 블로팅).

`MALLOC_ARENA_MAX=2`로 arena 수를 제한해 배포했습니다.

- 스테이징: 3시간 후 412MB → 적용 후 약 260MB로 개선 확인
- 프로덕션: 상승 속도가 절반으로 떨어짐(시간당 약 250MB → 약 130MB)

속도는 분명히 줄었습니다. 하지만 **방향은 그대로였습니다.** 24시간 뒤 RSS는 여전히 위험 수준까지 올라 `ENOMEM`이 재발했습니다. 단편화는 증상을 완화했을 뿐 원인이 아니었습니다.

### 가설 C — heap도 천천히 오르고 있다

단편화를 억제한 뒤 로그를 다시 보니 heap 사용량도 140~310MB까지 천천히 오르고 있었습니다. 가설 A에서 heap이 정상이라고 본 건 단편화에 가려진 착시였습니다. 처음에는 별도의 heap 누수를 의심했습니다. 다만 이는 독립적인 누수라기보다 GC 주기와 관련된 현상에 가까웠고 원인은 다음 가설에서 드러났습니다.

### 가설 D — GC를 게으르게 만든 잘못된 튜닝

여기서 과거의 잘못된 처방을 발견했습니다. 이전에 OOM이 났을 때 메모리가 부족하다고 판단해 `--max-old-space-size=4096`을 올려둔 설정이 있었습니다.

그런데 인과가 거꾸로였습니다. **OOM의 원인은 메모리 부족이 아니라 RSS 누수였습니다.** heap limit을 높게 잡으면 V8이 GC를 트리거하는 임계값도 함께 올라가 수거 시점이 늦어지고 결과적으로 RSS 누적이 가속됩니다. limit을 2048로 낮춰 GC를 더 적극적으로 돌게 했습니다.

> **OOM이 났을 때 heap을 올리는 건 종종 증상을 키우는 처방입니다.** heap limit을 만지기 전에 RSS 패턴부터 봐야 합니다.

## 3. 진짜 원인

가설 A~D를 거치며 범위는 네이티브 레이어에서 메모리를 충분히 반환하지 않는 무언가로 좁혀져 있었습니다. 마지막으로 주목한 대상은 이미지 처리에 쓰던 `node-canvas`였습니다.

`node-canvas`는 Cairo 기반 C++ 바인딩입니다. 장시간 부하가 누적되는 환경에서 작업이 끝난 뒤에도 RSS가 작업당 수십 MB씩 계속 증가하는 현상이 관찰됐습니다. 동일한 워크로드를 `@napi-rs/canvas`(Rust/skia 기반, N-API 구현)로 교체하자 RSS 우상향이 완전히 사라졌습니다. 라이브러리 내부 구현까지 단정할 수는 없었지만 운영 환경에서 node-canvas는 문제를 재현했고 교체 후 사라졌다는 점에서 원인을 충분히 특정할 수 있었습니다.

heap이 멀쩡했던 것도, external 지표로 깔끔하게 안 잡혔던 것도 누수가 V8 heap 바깥의 네이티브 영역에 있었기 때문입니다. `@napi-rs/canvas`는 API도 거의 호환돼 마이그레이션 비용이 작았습니다.

| | `node-canvas` | `@napi-rs/canvas` |
|---|---|---|
| 엔진 | C++ (Cairo/Pango) | Rust (skia) |
| 네이티브 빌드 | 필요 (node-gyp) | 불필요 (prebuilt binary) |
| 시스템 의존성 | libcairo, libpango 등 | 자체 번들, 없음 |
| Canvas 2D API | 지원 | 지원 (동일) |
| 장시간 RSS 거동 | 우상향 재현 (문제 지점) | 수평 유지 |

## 4. 최종 배포와 검증

원인을 확정한 뒤 세 가지를 함께 배포했습니다. 각각 역할이 다릅니다.

1. `MALLOC_ARENA_MAX=2` 유지 — 단편화 억제(보조)
2. `--max-old-space-size` 축소 — GC 적극화(보조)
3. **`node-canvas` → `@napi-rs/canvas` 교체 — 핵심 원인 제거**

검증에서 신경 쓴 건 배포 직후 숫자에 속지 않는 것이었습니다. 이전 배포에서도 직후 RSS는 정상으로 보였다가 6시간 뒤 두 배가 된 적이 있었습니다. 그래서 **최소한 정오 피크를 통과한 뒤까지** 관찰하는 걸 검증 기준으로 잡았습니다.

- 배포 직후 메모리 사용률 40% → 6~7%로 급락 후 수평 유지
- 새벽 고부하 구간(평소 두 번째로 많은 처리량)에서도 RSS 완전 수평
- 정오 피크에서 **기존 최대 대비 +55% 처리량**을 소화하고도 RSS 수평 유지
- 누수 패턴 완전 소멸 확인

즉 처리량 증가와 RSS 증가가 더 이상 비례하지 않았습니다.

## 5. 회고

**RSS만 오르면 애플리케이션 코드부터 의심하지 않습니다.** heap/external이 정상이라면 그 아래(glibc 단편화, 네이티브 라이브러리의 메모리 반납)를 봐야 합니다. 이 사고의 순서를 잡는 데 가설 A~B를 다 써버렸지만 다음엔 더 빨리 내려갈 수 있습니다.

**OOM에 heap 증설로 대응하는 건 위험합니다.** 원인이 누수면 heap limit 증가는 GC를 게으르게 만들어 누수를 가속합니다. 인과를 거꾸로 짚은 과거의 처방이 문제를 키우고 있었습니다.

**검증 기준은 배포 직후가 아니라 피크 통과 후입니다.** 메모리 문제는 시간 축 위에서만 드러납니다. 직후 스냅샷으로 해결을 선언하는 순간 같은 함정을 반복합니다.

부수적으로 근본 원인이 사라지자 검토 중이던 자동 재시작 스케줄러 도입이 불필요해졌습니다. 누수를 전제로 한 우회책들을 함께 정리할 수 있었던 것도 성과였습니다.
