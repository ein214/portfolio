# Bull Queue 워터마크 장애, 처리 능력은 멀쩡한데 31분을 기다린 이유

> 파일 다운로드에 워터마크를 입히는 백그라운드 큐에서, 개별 작업은 1~5초면 끝나는데 사용자 대기 시간이 31분까지 치솟는 장애가 났습니다. "처리 속도"가 아니라 "처리 용량 설계"가 원인이었던 이 문제를 추적하고, 이후 stalled job 중복 처리·graceful shutdown·오토스케일링 오작동까지 큐 운영 전반을 손본 기록입니다.

## TL;DR

- **장애**: 워터마크 큐 대기 시간이 최대 약 31분까지 누적. 개별 처리 시간은 1~5초로 정상이었다.
- **원인**: Bull Queue의 `concurrency`를 설정하지 않아 기본값 1로 동작 → 인스턴스당 동시 1개씩만 처리. 유입량이 처리 용량의 약 3.5배에 달하자 큐가 백로그로 쌓였다.
- **핵심 통찰**: "작업이 느린 것"과 "큐가 밀리는 것"은 완전히 다른 문제다. 처리 시간과 대기 시간을 분리해서 로깅하지 않으면 장애 원인을 거꾸로 짚게 된다.
- **확장**: concurrency 조정으로 끝내지 않고 lock/stalled 정책, 중복 처리 방어, graceful shutdown, 오토스케일링 지표 교체까지 큐 운영을 재정비했다.

---

## 1. 장애: 빠른 작업, 느린 응답

워터마크 큐에서 사용자 다운로드 대기 시간이 1시간가량 비정상적으로 길어지는 장애가 발생했습니다. 모니터링을 보니 기묘했습니다. **개별 작업의 실제 처리 시간은 1~5초로 완전히 정상**이었습니다. 그런데 최대 레이턴시는 약 31분이었습니다.

처리는 빠른데 응답이 느렸습니다. 이 간극의 정체는 **큐 대기 시간**이었습니다. 작업이 큐에 들어와서 워커가 집어들기까지 최대 31분을 기다리고 있었습니다.

원인은 용량 설계에 있었습니다. Bull Queue의 `concurrency` 옵션을 명시하지 않아 기본값 1로 돌고 있었고, 인스턴스가 여러 대여도 각 인스턴스가 동시에 1개씩만 처리하는 구조였습니다. 평상시엔 문제가 없다가 트래픽 피크에 유입량이 처리 용량의 약 **3.5배**로 뛰자 큐가 감당하지 못하고 백로그가 쌓였습니다.

> 교훈 하나. **큐 장애를 만나면 "작업이 느려졌나"보다 "큐가 밀렸나"를 먼저 봅니다.** 둘은 원인도 해법도 다릅니다. 작업이 느리면 작업을 최적화해야 하고, 큐가 밀리면 처리 용량(concurrency·인스턴스 수)을 늘려야 합니다.

## 2. 용량 재설계 — concurrency를 올리되, 경합을 계산하고 올린다

가장 직접적인 처방은 concurrency를 올리는 것입니다. 하지만 이 워커는 **단독으로 도는 큐가 아니었습니다.** 같은 컨테이너·같은 Redis를 6개의 큐가 공유하고 있었고, 그중 몇 개는 헤드리스 브라우저(Chromium) 기반이라 메모리를 크게 먹었습니다.

그래서 concurrency를 무작정 올리지 않고 두 가지를 먼저 따졌습니다.

**작업 성격.** 워터마크 작업은 외부 SDK·스토리지·DB 호출이 대부분인 I/O bound 작업입니다. CPU를 오래 점유하지 않으므로 CPU bound 작업보다 동시성을 공격적으로 올려도 됩니다. 게다가 처리 파일의 90% 이상이 소용량이라 메모리 압박도 낮았습니다.

**리소스 경합.** concurrency를 올렸을 때의 최대 메모리 점유를 계산하고, 같은 컨테이너의 Chromium 기반 큐들이 동시에 피크를 칠 때의 합산 메모리가 가용 한도 안에 들어오는지 확인했습니다. 즉 "이 큐만"이 아니라 "이 컨테이너 전체"의 메모리 예산으로 판단했습니다.

이 계산을 근거로 concurrency를 단계적으로 올려 이론 처리량을 한 자릿수 배수로 끌어올렸습니다(이론).

## 3. lock과 stalled — 설정 하나의 우선순위가 만든 함정

concurrency만 올리면 끝이 아니었습니다. 대용량 파일은 처리에 수십 초~1분 이상 걸리는데, 이 경우 Bull의 lock·stalled 메커니즘이 작업을 망가뜨릴 수 있습니다.

Bull은 워커가 작업을 잡으면 lock을 걸고, `lockDuration` 동안 응답이 없으면 "워커가 죽었다"고 판단해 **stalled** 상태로 만들어 다른 워커에게 재할당합니다. 문제는 정상 처리 중인 대용량 작업이 이 시간을 넘기면 살아있는데도 stalled로 찍혀 **같은 작업이 중복 실행**된다는 점입니다.

여기서 가장 까다로웠던 건 **설정 우선순위**였습니다. 작업을 큐에 넣는 쪽(producer)과 큐를 소비하는 쪽(worker)이 서로 다른 서버였는데, 일부 옵션은 producer가 작업을 추가할 때 준 값이 worker의 큐 설정을 덮어쓰고 일부는 그렇지 않았습니다.

| 설정 | producer가 덮어쓰는가 | 결과 |
|---|---|---|
| `lockDuration` | O (작업 옵션 우선) | producer가 짧게 준 값이 그대로 적용 → **의도한 lock 시간이 무력화** |
| `stalledInterval` | X (큐 설정만 유효) | worker 설정대로 동작 |
| `maxStalledCount` | X (큐 설정만 유효) | worker 설정대로 동작 |

`lockDuration`을 worker 쪽에서 길게 잡아놨는데도 producer가 작업마다 짧은 값을 넘기고 있어서 worker의 설정이 통째로 무시되고 있었습니다. **producer 쪽에서 해당 옵션을 제거해 worker의 큐 설정에 위임**하는 것으로 해결했습니다. 같은 옵션이라도 어느 레이어에서 주느냐에 따라 효력이 다르다는 걸 표로 정리해두지 않았으면 놓쳤을 함정입니다.

`lockDuration`에 대한 오해도 정리해둡니다. 이 값을 길게 잡는다고 해서 모든 작업이 그만큼 오래 점유되는 게 아닙니다. Bull은 정상 처리 중이면 `lockDuration/2` 간격으로 lock을 자동 갱신하고 작업이 끝나면 즉시 해제합니다. 즉 이 값은 "정상 작업의 점유 시간"이 아니라 "워커가 비정상 종료했을 때 얼마나 기다렸다가 재할당할지"의 안전 마진입니다.

## 4. stalled 중복 처리 방어 — 끄려고 했더니 더 꼬였다

별도 사건으로, 작업이 stalled로 빠지면서 유실되는 문제가 있었습니다. 처음엔 단순하게 접근했습니다. `stalledInterval`을 0으로 설정해 stalled 체크 자체를 꺼버리면 되지 않을까 생각했습니다.

**틀렸습니다.** `stalledInterval=0`은 "Bull이 stalled 여부를 체크하는 주기"를 끄는 것이지 "작업이 stalled 상태가 되는 것"을 막는 게 아니었습니다. lock이 만료되면 작업은 여전히 stalled가 되는데, 체크를 꺼두면 Bull이 이를 **즉시 감지하지 못해** 오히려 재처리가 한참 늦어지거나 방치됩니다.

결국 stalled를 막는 게 아니라 **stalled를 안전하게 처리하는** 방향으로 틀었습니다. 핵심은 멱등성입니다.

- 작업 처리 단계에서 중간 결과(생성된 파일의 위치 등)를 미리 저장해둡니다.
- `onStalled` 핸들러에서 `isCompleted()`로 실제 완료 여부를 확인합니다. stalled로 찍혔어도 처리 단계가 정상적으로 끝났다면 완료로 간주할 수 있습니다.
- 완료됐으면 `moveToCompleted`, 아니면 `moveToFailed`로 명시적으로 정리하고, 기존 완료/실패 핸들러를 재사용하도록 이벤트를 직접 발생시킵니다.
- 이 핸들러 전체를 try-catch로 감쌌습니다. 여기서 처리되지 않은 에러가 새면 워커 프로세스가 재시작되는 연쇄 장애로 번지기 때문입니다.

> 교훈. **재시도가 자동으로 일어나는 시스템에서는 "막기"보다 "여러 번 실행돼도 괜찮게 만들기"가 정답인 경우가 많습니다.** Bull은 stalled 작업을 자동 재시도하므로 중복 실행을 전제로 멱등성을 보장하는 쪽이 훨씬 견고합니다.

## 5. 배포 중 작업 유실 방어 — Graceful Shutdown

워커 인프라를 옮기는 과정에서 또 하나의 구멍이 보였습니다. 컨테이너가 교체될 때 처리 중이던 작업이 강제 종료되면 stalled로 빠진다는 점이었습니다.

SIGTERM을 받았을 때 안전하게 마무리하도록 종료 훅을 구현했습니다.

1. 새 작업 수신을 중단합니다 — `queue.pause(true)`. 여기서 인자 `true`가 중요합니다. 인자 없는 `pause()`는 Redis 큐 전체를 멈춰 **다른 워커까지** 영향을 주지만, `pause(true)`는 현재 워커만 멈춥니다.
2. `queue.close()`로 처리 중인 작업이 끝날 때까지 블로킹 대기한 뒤 연결을 닫습니다.
3. 인프라 쪽에서는 강제 종료(SIGKILL)까지의 대기 시간을 최대로 늘려, 처리 중 작업이 마무리될 시간을 벌어줍니다.

단, 이 안전망에도 전제가 있습니다. 작업 하나의 처리 시간이 강제 종료 대기 한도를 넘으면 결국 잘립니다. 그래서 "graceful shutdown을 구현했으니 안전하다"가 아니라 "작업 시간이 종료 대기 한도 안에 들어오는가"를 함께 확인해야 합니다. 넘는다면 작업을 더 잘게 쪼개는 구조 개선이 별도로 필요합니다.

## 6. 모니터링과 오토스케일링 — 지표를 잘못 보면 스스로 폭발한다

장애를 겪은 뒤 가장 먼저 한 일은 **대기 시간과 처리 시간을 분리해서 로깅**하는 것이었습니다. `현재 시각 - 작업 등록 시각`으로 대기 시간을, `처리 시작 ~ 반환`으로 처리 시간을 따로 찍었습니다. 이렇게 해두면 다음 장애 때 "작업이 느린 건지, 큐가 밀린 건지"를 즉시 구분할 수 있습니다. 1절의 31분짜리 혼란을 두 번 겪지 않기 위한 장치입니다.

큐 적체 알람도 붙였습니다. 다만 앱 내부에서 직접 알림을 쏘는 방식은 **앱이 죽으면 알림도 같이 죽는** 약점이 있습니다. 그래서 앱 내부 알림(상세 메시지용)과 앱 외부 모니터링(앱 다운 자체를 감지)을 함께 두는 하이브리드로 갔습니다. 외부 모니터링에서는 메트릭 발행이 끊기는 것 자체를 "앱이 죽었을 수 있음" 신호로 잡도록 구성했습니다.

여기서 멀티 인스턴스 특유의 문제가 하나 있었습니다. 인스턴스가 여러 대인데 모두 같은 주기로 메트릭을 발행하면 중복이 됩니다. Redis의 원자적 연산(`SET key value EX NX`)으로 경량 분산 락을 걸어 매 주기 한 대만 발행하도록 했습니다. TTL을 발행 주기보다 약간 짧게 잡아서 특정 인스턴스가 락을 독점하지 않고 매번 새로 경쟁하게 했고, 인스턴스가 죽어도 TTL이 지나면 자동 해제되어 데드락이 없습니다. 이건 엄밀한 분산 락이 아니라 "한 번쯤 중복되거나 빠져도 괜찮은 작업"에 맞는 경량 방식이라는 한계도 함께 기록해뒀습니다.

마지막으로, 오토스케일링이 **스스로 폭발한** 사건이 있었습니다. 인프라를 옮기면서 기존 스케일링 정책을 그대로 이식했는데, 알람이 보고 있던 지표가 새 환경의 실제 부하와 1:1로 대응하지 않았습니다. 지표가 줄지 않으니 알람이 계속 켜진 채로 쿨다운마다 인스턴스를 1대씩 추가했고 결국 상한까지 늘어났습니다.

두 가지를 고쳤습니다.

- **지표를 작업 성격에 맞게 교체.** 이 워커의 실제 부하는 CPU 집약적인데 엉뚱한 네트워크 지표를 보고 있었습니다. 부하와 상관관계가 높은 CPU 사용률로 바꿨습니다.
- **scale-up은 민감하게, scale-down은 보수적으로.** 이 워크로드는 CPU 스파이크가 짧게 치는 패턴이라, scale-up 판정을 오래 기다리면 이미 큐가 밀린 뒤입니다. 반대로 scale-down은 산발적 유입에 일시적으로 한가해 보일 때 성급히 줄이면 곧 다시 모자라므로 보수적으로 유지했습니다.

## 7. 회고

큐 운영에서 반복해 확인한 원칙들입니다.

**처리 시간과 대기 시간은 반드시 분리해서 봅니다.** 둘을 뭉뚱그리면 "빠른 작업이 31분 걸리는" 모순 앞에서 원인을 거꾸로 짚습니다. 모니터링 설계부터 이 둘을 나눠야 합니다.

**같은 옵션이라도 어느 레이어에서 주느냐에 따라 효력이 다릅니다.** producer/worker, 작업 옵션/큐 설정의 우선순위를 표로 정리하지 않으면 분명히 설정했는데 무시당하는 함정에 빠집니다.

**자동 재시도가 있는 시스템은 중복 실행을 전제로 멱등하게 만듭니다.** stalled를 끄려는 시도는 대개 더 꼬입니다. 막는 대신 안전하게 처리합니다.

**이식한 설정은 새 환경에서 재검증합니다.** lock 옵션도 오토스케일링 지표도, 옛 환경에서 멀쩡했던 설정이 새 환경에서 장애의 원인이 됐습니다. 환경이 바뀌면 전제도 바뀝니다.
