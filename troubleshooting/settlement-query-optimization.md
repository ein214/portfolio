# 천만 건 정산 데이터 조회 개선, 인덱스도 파티셔닝도 안 듣던 이유

> 누적 천만 건을 넘긴 정산 데이터에서 한 달치 조회/엑셀 다운로드가 사실상 불가능해진 문제를 해결한 기록입니다. VACUUM/REINDEX, 파티셔닝까지 원본 테이블을 튜닝하는 정공법을 차례로 시도했지만 모두 실패했고 결국 조회 전용 비정규화 테이블과 메시지 큐 기반 동기화로 읽기 경로를 분리해 풀었습니다. "왜 정석 튜닝이 안 통했는가"가 이 글의 핵심입니다.

## TL;DR

- **문제**: 정산 테이블이 800만 건(연관 이용 로그 테이블은 1,100만 건)을 넘기면서, 특정 조건(대량 데이터를 가진 정산 유형 × 성수기 한 달) 조회 실행 시간이 약 19초까지 치솟았다. 엑셀 다운로드는 사실상 받을 수 없었다.
- **1차 시도 — 테이블 튜닝**: 인덱스 정리, REINDEX, VACUUM/ANALYZE, dead tuple, bloat은 해소됐지만 문제의 조회는 그대로.
- **2차 시도 — 파티셔닝**: 월별 파티셔닝. 파티션 프루닝은 정상 동작했지만, 한 파티션 안에 160만 건 + 다중 조인이 겹치자 옵티마이저가 인덱스를 포기하고 해당 파티션을 통째로 Seq Scan했다.
- **결론 — 읽기 경로 분리**: 조인이 전혀 필요 없는 조회 전용 Flat Table을 만들고, 정산 생성/확정 시 메시지 큐로 동기화. 조회는 단일 테이블 스캔으로 단순화.

---

## 1. 문제: 정석대로 인덱스를 걸어도 안 걸리는 쿼리

정산 데이터가 누적 800만 건을 넘기고 연관된 이용 로그 테이블은 1,100만 건을 넘기면서 내부 운영의 한 달치 정산 조회와 엑셀 다운로드가 한계에 부딪혔습니다. 특히 데이터가 폭증하는 정산 유형이 하나 있었습니다. 이 유형은 배치 한 번에 십만 건 단위로 증식했습니다. 성수기 한 달이면 이 유형 하나만 약 166만 건에 달했습니다(같은 달 전체가 약 185만 건이었으니 대부분이 이 유형이었습니다).

이 조건으로 조회하면 실행 시간이 약 19초였습니다. 운영자가 엑셀을 누르면 받지 못하고 타임아웃이 나는 수준이었습니다(한 달 데이터를 뽑기 위해 15일씩 나눠서 뽑음).

처음엔 전형적인 "인덱스 안 탄 쿼리" 문제로 봤습니다. 그런데 조건에 따라 옵티마이저가 **어떤 인덱스도 쓰지 않고** 풀스캔을 선택했습니다. 왜 그런지 이해하는 데서부터 시작해야 했습니다.

## 2. 1차 시도 — 원본 테이블 튜닝 (실패)

먼저 테이블 자체의 상태를 점검했고 몇 가지 문제가 발견됐습니다.

* **autovacuum이 충분히 수행되지 않고 있었습니다.** 약 800만 행 규모의 테이블에서 한 달 이상 autovacuum이 실행되지 않은 상태였고, dead tuple 비율도 6% 수준까지 증가해 있었습니다. 조회 성능 저하 가능성이 있는 상태였습니다.
* **인덱스 bloat가 누적돼 있었습니다.** 테이블 데이터 크기는 약 2.4GB였지만 인덱스를 포함한 전체 크기는 약 4.8GB에 달했습니다. INSERT·UPDATE가 많은 테이블 특성상 오래된 인덱스 엔트리가 누적되며 비효율이 발생한 상태였습니다.
* **사용되지 않는 대형 인덱스가 존재했습니다.** 인덱스별 사용 횟수를 확인해 보니 수백 MB를 차지하면서도 실제 조회에서는 거의 사용되지 않는 인덱스들이 있었습니다. 공간을 차지할 뿐 아니라 쓰기 비용까지 증가시키는 부채였습니다.

이에 먼저 원본 테이블에 대한 튜닝을 시도했습니다. 수동 VACUUM/ANALYZE를 수행해 통계를 최신화하고 사용되지 않는 인덱스를 정리했으며 REINDEX를 통해 비대해진 인덱스를 재구성했습니다. VACUUM만으로는 누적된 인덱스 bloat 자체를 해소할 수 없기 때문에 REINDEX가 필요했습니다.

이러한 정비를 통해 테이블 상태는 개선됐지만 핵심 조회의 성능은 기대만큼 나아지지 않았습니다. 즉, 병목은 dead tuple이나 인덱스 bloat가 아니었습니다. 원인은 더 아래 조회 구조 자체에 있었습니다.

## 3. 2차 시도 — 파티셔닝 (부분 성공, 본질은 실패)

다음 가설은 "테이블이 너무 크니 조회 범위를 줄이자"였습니다. 정산 데이터는 월 단위로 조회하는 경우가 대부분이었기 때문에 생성일 기준 월별 파티셔닝이 자연스러운 선택이었습니다. 조회 시 해당 월 파티션만 읽도록 만들어 전체 스캔 범위를 줄이는 것이 목적이었습니다.

파티셔닝 자체는 정상적으로 동작했습니다. WHERE 조건에 생성일이 포함되면 파티션 프루닝이 적용돼 필요한 월의 파티션만 읽도록 실행 계획이 생성됐습니다.

하지만 핵심 조회의 성능은 기대만큼 개선되지 않았습니다.

원인을 확인하기 위해 EXPLAIN ANALYZE로 실제 실행 계획을 분석했습니다. 특정 성수기 월에는 단일 파티션 안에도 수십만~백만 건 규모의 데이터가 존재했고 조회 결과를 사람이 읽을 수 있는 형태로 만들기 위해 여러 연관 테이블과의 조인이 필요했습니다. 정산 한 건을 표현하기 위해 결제/주문/상품/계약 계열의 데이터를 함께 조회해야 했습니다.

실행 계획에서 확인한 사실은 다음과 같았습니다.

* 파티션 프루닝은 정상 동작했습니다. 조회 대상은 하나의 월 파티션으로 제한됐습니다.
* 하지만 해당 파티션은 인덱스 스캔이 아닌 Seq Scan(순차 전체 스캔)으로 읽고 있었습니다.
* 그 위로 여러 테이블에 대한 Hash Join이 연쇄적으로 수행됐고 전체 실행 시간은 약 19초에 달했습니다.

옵티마이저 입장에서 수십만~백만 건 규모의 데이터와 다수의 조인이 결합된 상황에서는, 인덱스를 이용한 랜덤 액세스보다 데이터를 한 번에 읽어 Hash Join을 수행하는 편이 더 비용이 낮다고 판단할 수 있습니다. 실제로 이 경우에도 인덱스를 사용하지 않는 실행 계획이 선택됐습니다.

이때 얻은 교훈은 명확했습니다.

> **파티셔닝은 조회 범위를 줄이는 기법이지 조인 비용을 줄이는 기법은 아닙니다.**
>
> 병목이 데이터 양이 아니라 조인 복잡도에 있을 경우 파티셔닝만으로는 문제를 해결할 수 없습니다. 또한 대량 데이터와 다중 조인이 결합된 상황에서는 인덱스를 추가하는 것보다 Seq Scan과 Hash Join이 더 효율적인 실행 계획이 될 수 있다는 사실을 비용 모델 관점에서 받아들여야 했습니다.


## 4. 결론 — 문제를 쿼리에서 데이터 모델로 옮기다

1·2차 시도를 통해 한 가지가 분명해졌습니다. 조회가 느린 진짜 원인은 개별 쿼리가 아니라 읽을 때마다 여러 테이블을 조인해야 하는 구조 자체였습니다.

스타트업 환경에서는 새로운 인프라를 도입하기 전에 기존 자원을 최대한 활용하는 것이 중요했습니다. 그래서 먼저 인덱스 정비와 파티셔닝처럼 추가 비용 없이 적용할 수 있는 개선을 우선 검토했습니다. 하지만 두 번의 시도를 거치며 문제의 본질이 쿼리 튜닝이 아니라 데이터 모델에 있다는 결론에 도달했습니다.

결국 문제를 쿼리 레벨이 아니라 데이터 모델 레벨로 옮기기로 했습니다. 조회 시점마다 조인을 수행하는 대신 쓰기 시점에 필요한 정보를 미리 결합한 읽기 모델을 별도로 유지하는 방식입니다. 정규화를 일부 포기하더라도 조회 성능을 확보하는 것이 더 중요하다고 판단했습니다.

설계는 다음과 같았습니다.

### 조회 전용 Flat Table

정산 한 건을 표현하는 데 필요한 정보를 하나의 테이블에 비정규화해 저장했습니다. 조회 시 조인을 수행하지 않고 단일 테이블만 읽도록 만들었습니다.

또한 거래 시점의 정보가 중요하다는 도메인 특성을 고려해 원본 데이터의 이름이 변경되더라도 과거 데이터를 역으로 수정하지 않는 정책을 선택했습니다.

### 유형별 테이블 분리

데이터 양과 조회 패턴이 크게 다른 정산 유형은 별도 테이블로 분리했습니다. 일부 대용량 데이터가 전체 조회 성능에 영향을 주지 않도록 읽기 모델 자체를 분리했습니다.

### 읽기/쓰기 DB 분리

원본 데이터를 관리하는 쓰기 DB와 조회를 담당하는 읽기 DB를 분리했습니다. 이를 통해 대용량 조회나 배치 작업이 정산 생성 경로에 영향을 주지 않도록 했습니다.

### 메시지 큐 기반 동기화

정산 생성 및 상태 변경 시 이벤트를 발행하고 별도의 컨슈머가 이를 수신해 조회용 데이터를 갱신하도록 구성했습니다.

```text
정산 생성/변경 (쓰기 DB)
        │
        ▼
     메시지 큐
        │
        ▼
컨슈머
        │
        ▼
조회용 Flat Table (읽기 DB)
        │
        ▼
운영 시스템 / 사용자 조회
```

읽기 모델이 완성된 이후에는 대부분의 조회가 조인 없이 단일 테이블에서 처리됐습니다.

## 5. 곁가지지만 중요했던 것 — 에러 데이터의 구조화

정산 시스템에서는 성능만큼 정확성이 중요했습니다.

실제 운영 과정에서 발생하는 문제의 상당수는 시스템 버그가 아니라 잘못된 매핑, 누락된 데이터, 계약 정보 불일치와 같은 운영상의 오류였습니다. 기존에는 원본 데이터를 직접 수정하는 방식으로 대응했고, 변경 이력을 추적하기 어렵다는 문제가 있었습니다.

그래서 오류 자체를 구조화하기로 했습니다. 발생 가능한 오류를 코드화하고 별도의 로그 테이블에 기록하도록 만들었습니다.

수정 역시 원본 데이터를 직접 변경하지 않고 동일한 이벤트 기반 구조를 따르도록 했습니다.

```text
운영 도구
    │ 수정 요청
    ▼
메시지 큐
    ▼
원본 데이터 수정
    ▼
동기화 이벤트 발행
    ▼
조회용 데이터 반영
```

결국 조회를 위해 만든 비동기 동기화 구조를 수정 경로에도 재사용함으로써 데이터 변경 흐름을 일관된 방식으로 관리할 수 있었습니다.

## 6. 회고

세 가지를 남깁니다.

### 느린 쿼리를 만나면 먼저 문제가 있는 층을 구분합니다

dead tuple, 인덱스 bloat, 통계 정보와 같은 문제는 튜닝으로 해결할 수 있습니다. 하지만 병목이 조인 구조나 데이터 모델에 있다면 인덱스를 추가해도 옵티마이저는 결국 Seq Scan을 선택합니다.

인덱스 정비와 파티셔닝을 먼저 시도한 이유도 여기에 있었습니다. 비용이 거의 들지 않는 방법부터 검증하고 구조 변경이 정말 필요한지 확인하고 싶었습니다.

결과적으로 두 번의 실패는 불필요한 시행착오가 아니라 문제의 본질이 데이터 모델에 있다는 사실을 확인하는 과정이었습니다.

### 파티셔닝은 조인 비용을 해결해 주지 않습니다

파티셔닝은 조회 범위를 줄이는 도구입니다. 실제로 파티션 프루닝은 정상적으로 동작했지만 단일 파티션 내부의 대량 데이터와 다중 조인이라는 병목 자체는 그대로 남아 있었습니다.

도구마다 해결할 수 있는 문제가 다르며 효과 범위를 정확히 이해하는 것이 중요하다는 점을 배웠습니다.

### 읽기와 쓰기는 서로 다른 모델을 가질 수 있습니다

쓰기 모델은 정합성이 중요하고 읽기 모델은 성능이 중요합니다.

두 요구를 하나의 모델로 해결하려 하기보다 각각의 목적에 맞는 모델을 분리하고 비동기로 연결하는 편이 더 단순하고 견고할 수 있습니다.

천만 건 규모의 데이터에서는 이러한 분리가 결국 가장 큰 성능 개선으로 이어졌습니다.
